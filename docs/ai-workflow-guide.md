# AI 辅助项目开发工作流指南

> 本文档记录了从零构建 `ai-task-manager` 项目的完整开发流程，可作为 AI 创建类似项目的标准化工作流参考。

## 概述

通过 AI（Claude Code）从零构建一个 Python CLI 工具的全流程，涵盖：需求澄清 → 架构设计 → 实现计划 → 子智能体驱动开发 → 测试驱动开发 → 代码审查 → 发布部署。

**项目规模：** ~900 行 Python 代码，44 个测试，14 次提交
**耗时：** 单会话连续完成
**核心工具：** Claude Code + Superpowers 技能体系

---

## 第一阶段：需求澄清（Brainstorming）

### 流程

不要直接开始编码，先通过结构化对话厘清需求。

### 关键问题清单

| 问题 | 目的 | 示例 |
|------|------|------|
| 你想构建什么？ | 获取项目全景 | "管理开发项目的任务系统" |
| 项目 vs 代码目录的关系？ | 确定数据模型 | "每个项目对应一个本地代码目录" |
| 需要初始化新项目吗？ | 确定功能范围 | "需要创建目录 + git init" |
| 子智能体如何分配？ | 确定自动化程度 | "自动分配并执行" |
| 交互方式？ | 确定 UX 风格 | "对话 + CLI 混合" |
| HTML 展示形态？ | 确定前端风格 | "看板式" |
| 数据存储格式？ | 确定技术选型 | "JSON 文件" |

### 原则

- **一次问一个问题**，不要一次性抛出多个问题
- **优先用选择题**，降低用户回答成本
- **范围评估**：如果需求涉及多个独立子系统，先分解再逐个讨论
- **YAGNI**：主动剔除不必要的功能

## 第二阶段：架构设计（Design）

### 流程

需求明确后，提出 2-3 种方案并推荐其一。

### 设计方案示例

```
方案 1: Shell/文件方案（轻量）
方案 2: Python CLI 工具（推荐）
方案 3: MCP Server 方案
```

### 设计文档应包含

1. **目录结构** — 所有文件及其职责
2. **数据模型** — 核心数据结构（字段、类型、验证规则）
3. **命令设计** — CLI 命令及参数
4. **UI/展示设计** — HTML 模板、视觉风格
5. **存储架构** — 文件组织、Token 优化策略
6. **依赖清单** — 第三方库

### 关键决策

- **数据模型用 dataclass** — Python 原生支持，简洁且类型安全
- **存储用 JSON 文件** — 结构明确、人类可读、Git 友好、按需加载
- **CLI 用 Click** — 成熟稳定，支持交互式 prompt
- **模板用 Jinja2** — HTML 渲染，自动转义防 XSS

### 增量确认

每展示一个设计部分就问"这部分 OK 吗？"，避免全部设计完才发现方向不对。

## 第三阶段：实现计划（Writing Plans）

### 流程

将设计文档转化为可执行的、原子化的任务列表。

### 任务分解原则

每个 Task 应该是：
- **独立可完成**：产出可测试的代码
- **小粒度**：每个步骤 2-5 分钟
- **包含完整代码**：不写"参考上文"，重复代码
- **TDD 流程**：先写测试 → 运行失败 → 实现 → 运行通过 → 提交

### 任务清单示例

```
Task 1: 项目脚手架（目录、依赖）
Task 2: 数据模型（Project + Task）
Task 3: 存储层（JsonStore）
Task 4: 项目 CRUD 命令
Task 5: 任务 CRUD 命令
Task 6: Agent 分配命令
Task 7: HTML 看板模板
Task 8: 报告生成命令
Task 9: 主入口整合
Task 10: 项目初始化能力
Task 11: 最终验证
```

### 计划文档结构

每个 Task 包含：
- **文件清单** — 创建/修改哪些文件
- **测试代码** — 完整的 pytest 代码
- **运行命令** — 确切的命令和预期输出
- **实现代码** — 完整的实现
- **提交命令** — git add + commit

## 第四阶段：子智能体驱动开发（Subagent-Driven Development）

### 核心模式

```
每个 Task = 1 个实现子智能体 + 1 个规范审查 + 1 个代码质量审查
```

### 工作流

```
┌─────────────────────────────────────────────────┐
│ 1. 派发实现子智能体（含完整 Task 描述 + 上下文）   │
│    → 子智能体可提问澄清                            │
│    → 实现、测试、提交、自审查                       │
│    → 报告状态（DONE / DONE_WITH_CONCERNS）         │
├─────────────────────────────────────────────────┤
│ 2. 派发规范审查子智能体                             │
│    → 读取实际代码，逐条核对需求                      │
│    → ✅ 通过 / ❌ 发现问题                         │
│    → 如发现问题 → 回到步骤 1 修复 → 重新审查        │
├─────────────────────────────────────────────────┤
│ 3. 派发代码质量审查子智能体                          │
│    → 检查命名、可读性、错误处理、测试质量              │
│    → ✅ 批准 / ⚠️ 警告 / ❌ 阻止                   │
│    → 如发现问题 → 回到步骤 1 修复 → 重新审查         │
├─────────────────────────────────────────────────┤
│ 4. 标记 Task 完成，进入下一个 Task                   │
└─────────────────────────────────────────────────┘
```

### 子智能体提示模板

#### 实现子智能体

```
You are implementing Task N: [task name]

## Task Description
[完整的 Task 描述，包含所有代码]

## Context
[场景说明：这个 Task 在项目中的位置、依赖关系]

## Before You Begin
如有疑问，现在问。澄清后再开始实现。

## Your Job
1. 实现 Task 描述的代码
2. 遵循 TDD
3. 验证测试通过
4. 提交
5. 自审查并报告
```

#### 规范审查子智能体

```
You are reviewing whether an implementation matches its specification.

## What Was Requested
[完整的 Task 需求]

## CRITICAL: Do Not Trust the Report
读取实际代码验证，不要相信实现者的报告。

检查：
- 缺少的需求
- 多余的功能
- 误解的需求
```

#### 代码质量审查子智能体

```
Review the implementation for quality.

检查：
- 命名、可读性、错误处理
- 测试质量
- 文件职责单一性
- 代码规模控制
```

### 关键原则

- **串行执行 Task**：不要并行派发多个实现子智能体，避免文件冲突
- **每次完整上下文**：给子智能体提供完整的 Task 描述，不要让它自己去读计划文件
- **问题必须回答**：子智能体提问时，必须先回答再让它继续
- **审查不可跳过**：不能跳过规范审查或代码质量审查
- **修复循环**：审查发现问题 → 实现者修复 → 重新审查，直到通过

## 第五阶段：跨平台兼容性处理

### 常见问题

| 问题 | 解决方案 |
|------|----------|
| Windows GBK 终端不支持 emoji | try/except UnicodeEncodeError 包装输出函数 |
| 文件路径分隔符 | 使用 `pathlib.Path` 跨平台路径 |
| Python 路径差异 | 使用 `python3 -m pytest` 而非直接 `pytest` |

### emoji 兼容工具示例

```python
# utils/console.py
import click

_EMOJI_MAP = {
    "✅": "[OK]", "❌": "[ERROR]", "📋": "[LIST]",
    "🟢": "[>]", "🟡": "[~]", "🔴": "[!]",
    "🔄": "[...]", "👀": "[REV]",
}

def echo(message="", **kwargs):
    try:
        click.echo(message, **kwargs)
    except UnicodeEncodeError:
        safe = message
        for em, repl in _EMOJI_MAP.items():
            safe = safe.replace(em, repl)
        click.echo(safe, **kwargs)
```

## 第七阶段：经验总结

### 什么做得好

1. **先设计再编码**：Brainstorming 阶段发现了多个隐含需求（--init 标志、混合交互模式），避免了返工
2. **子智能体隔离**：每个 Task 独立子智能体，上下文纯净，不会串线
3. **TDD 确保质量**：44 个测试覆盖了核心路径和边界情况
4. **规范审查发现了遗漏**：如 `data.pop()` 会修改输入 dict，`from_dict` 不可变性问题
5. **代码质量审查提升了标准**：如 `frozenset` 替代 `set`、`import` 位置规范

### 可以改进的

1. **Windows 兼容性应提前考虑**：emoji、编码问题在开发后期才修复
2. **任务 ID 用 UUID 而非顺序数字**：这个设计缺陷到最终审查才被发现
3. **错误处理模式应从一开始统一**：ClickException vs echo + return 的不一致需要后期统一修复
4. **测试夹具（fixture）应更早设计**：测试中 `monkeypatch` + `tmp_path` 的模式在 Task 4 才确立，前面的测试需要调整

### Token 消耗统计（参考）

| 阶段 | Token 消耗 | 说明 |
|------|-----------|------|
| Brainstorming | ~2K | 问答对话 |
| 设计文档 | ~3K | 写入文件 |
| 实现计划 | ~8K | 详尽的代码+测试 |
| 每个 Task 实现 | ~15-50K | 含上下文传递 |
| 每次代码审查 | ~15-40K | 含实际代码读取 |
| **总计** | **~300-500K** | 全流程 |

## 快速参考：AI 开发 Checklist

### 启动前
- [ ] 调用 brainstorming 技能
- [ ] 探索项目上下文
- [ ] 逐问澄清需求（一次一问）
- [ ] 提出 2-3 个方案并推荐
- [ ] 逐部分展示设计并确认

### 计划阶段
- [ ] 调用 writing-plans 技能
- [ ] 任务分解到原子粒度（每个 2-5 分钟）
- [ ] 每个任务含完整代码和测试
- [ ] 自检：无占位符、无类型不一致、全覆盖

### 执行阶段
- [ ] 调用 subagent-driven-development 技能
- [ ] 串行执行 Task
- [ ] 每个 Task: 实现 → 规范审查 → 代码审查
- [ ] 审查发现问题 → 修复 → 重新审查

### 收尾阶段
- [ ] 调用 finishing-a-development-branch 技能
- [ ] 全量测试通过
- [ ] 最终代码审查
- [ ] 跨平台兼容性检查
- [ ] 推送到远程 / 合并
